# 《打工人康复指南》MVP 界面草图方案

版本：v1.0  
更新日期：2026-03-30  
用途：MVP 页面布局与 UI 方向选择  
阶段：低保真线框方案

---

## 一、设计前提

本文件不解决高保真视觉风格，只解决以下问题：

- 页面怎么排
- 哪些信息放在哪
- 用户视线先看哪里
- 拖拽工具区怎么更像“游戏”
- 提示层和结算层如何不打断体验

目标是让页面一眼看起来像“微信小游戏”，而不是“普通小程序页面”。

统一前提：

- 竖屏
- 一屏完成核心操作
- 顶部信息、中部角色、底部工具栏的三段式结构不变
- MVP 只考虑 1 个关卡主界面 + 1 个提示弹层 + 1 个结算弹层

---

## 二、方案 A：经典小游戏版

### 适合场景

- 最稳妥
- 最容易在 Cocos 里直接搭
- 最像常见的轻度休闲小游戏
- 适合先做 MVP 验证

### 页面结构

```text
+--------------------------------------------------+
| Lv.1 键盘侠的觉醒              [?]               |
| 健康值 [########------] 50%                      |
|                                                  |
|                 角色主展示区                     |
|                                                  |
|              [胸前红色热区]                      |
|                                                  |
|                 打工人角色                       |
|                                                  |
|              [后背蓝色热区]                      |
|                                                  |
|                                                  |
| 反馈文案：这地方不是这么治的                     |
|                                                  |
|==================================================|
| [筋膜球]                      [弹力带]           |
| 放松                           强化              |
|                                                  |
+--------------------------------------------------+
```

### 布局比例建议

- 顶部信息区：15%
- 中部角色区：60%
- 底部工具栏：25%

### UI 特征

- 顶部是标准关卡信息
- 中间突出角色和热区
- 底部工具栏像“道具栏”
- 反馈文案浮在角色区和工具栏之间

### 优点

- 结构最清楚
- 开发最简单
- 用户理解成本最低
- 很适合后续扩展更多关卡

### 风险

- 视觉上会偏保守
- 如果美术弱一点，容易显得普通

---

## 三、方案 B：病例治疗台版

### 适合场景

- 想让游戏更有“诊疗感”
- 想强化“我是一个康复师，在处理病例”
- 页面更有主题感

### 页面结构

```text
+--------------------------------------------------+
| 病例 01：键盘侠圆肩驼背            提示 / 帮助    |
| 健康恢复度 50%                                    |
|--------------------------------------------------|
|                                                  |
|        ┌────────────────────────────┐            |
|        |        患者治疗台           |            |
|        |                            |            |
|        |      [胸前异常区域]        |            |
|        |         角色主体           |            |
|        |      [后背异常区域]        |            |
|        |                            |            |
|        └────────────────────────────┘            |
|                                                  |
| 当前判断：胸紧背弱                                |
|                                                  |
| 工具架：                                          |
| [筋膜球]   [弹力带]                               |
|                                                  |
+--------------------------------------------------+
```

### 布局特点

- 中间区域像一个“治疗面板”或“病例板”
- 工具区更像器械架
- 顶部标题不写传统关卡，更像病例编号

### UI 特征

- 有轻微“专业感”
- 比普通小游戏更有主题包装
- 很适合后续加入更多“病人档案”

### 优点

- 主题统一感更强
- 容易把医学逻辑包装成“解题”
- 分享截图会更有记忆点

### 风险

- 如果处理不好，会有点像“教育产品”
- 比方案 A 多一层容器设计，布局稍复杂

---

## 四、方案 C：夸张漫画对战版

### 适合场景

- 想更有梗、更像微信爆款轻游戏
- 想做得更夸张、更偏娱乐化
- 强调“治疗动作”而不是“知识讲解”

### 页面结构

```text
+--------------------------------------------------+
| [返回]        打工人康复指南        [提示]        |
| 恢复进度：50%                                    |
|                                                  |
|               Boss：歪掉的程序员                 |
|                                                  |
|                  [红色弱点]                       |
|                     \ | /                        |
|                      角色                         |
|                     / | \                        |
|                  [蓝色弱点]                       |
|                                                  |
|        正确操作会触发夸张反馈与文案               |
|                                                  |
|--------------------------------------------------|
| 可用技能：                                        |
| [筋膜球技能卡]   [弹力带技能卡]                   |
|                                                  |
+--------------------------------------------------+
```

### 布局特点

- 角色更像一个“敌人目标”
- 工具更像技能卡
- 文案、动画、反馈都可以更夸张

### UI 特征

- 更年轻
- 更娱乐化
- 更适合做搞笑传播

### 优点

- 传播感最强
- 更像“游戏”而不是“工具”
- 适合以后做多病人、多 Boss 关卡

### 风险

- 如果拿捏不好，会削弱“康复主题”
- 对文案和视觉表现要求更高

## 五、三版对比结论

| 方案 | 关键词 | 上手难度 | 开发难度 | 游戏感 | 推荐度 |
| ---- | ---- | ---- | ---- | ---- | ---- |
| A | 经典小游戏 | 最低 | 最低 | 中 | 高 |
| B | 病例治疗 | 低 | 中 | 中高 | 高 |
| C | 漫画对战 | 中 | 中高 | 最高 | 中高 |

如果目标是：

- 尽快做出 MVP：优先选 A
- 想保留“康复题材”的辨识度：优先选 B
- 想做得更像轻度爆款小游戏：优先选 C

---

## 六、我的建议

如果你现在是一个人推进，又希望尽快把 MVP 做出来，我最推荐：

`方案 A 的结构 + 方案 B 的主题包装`

也就是：

- 整体布局用方案 A，最稳
- 顶部文案、页面命名、部分容器风格借鉴方案 B

这样能同时得到：

- 够像游戏
- 不难做
- 不会太像普通小程序
- 康复主题也能立住

## 八、MVP 视觉方向建议

先不做高保真，但建议先定下面这些基础规则：

- 主背景用浅灰蓝或浅米色，不要纯白
- 工具栏用深色底，像游戏 HUD
- 红区和蓝区颜色一定要足够明显
- 角色主体要放大，占视觉中心
- 按钮要圆角、饱满，不能太像表单按钮
- 标题和反馈文案可以带一点漫画口吻

## 十、定稿方案：B 方案开发版

当前确定采用：

`方案 B：病例治疗台版`

目标不是做成严肃医疗产品，而是做成“有主题包装的轻游戏界面”。

核心原则：

- 看起来像在处理一个病例
- 但操作上仍然像微信小游戏
- 页面一眼能看懂，不做复杂层级
- 所有核心操作都在一屏完成

## 十一、主界面布局标注

建议以竖屏 `750 x 1334` 作为设计参考尺寸。

## 十二、模块说明

## 12.1 顶部标题区

### 包含内容

- 左侧：病例标题
- 右侧：提示按钮

### 文案建议

- 标题：`病例01：键盘侠的觉醒`
- 提示按钮：`提示`

### 设计建议

- 标题放在深灰蓝色条内
- 字体要比普通正文更有识别度
- 提示按钮做成圆角矩形按钮，不要做成纯图标，避免用户不点

## 12.2 状态区

### 包含内容

- 病人标签或状态标签
- 恢复百分比
- 进度条

### 文案建议

- 左侧标签：`键盘侠衰衰`
- 百分比：`50%`

### 设计建议

- 状态区高度不要太高，控制在一行半以内
- 进度条要够粗，保证手机上容易识别
- 进度条颜色建议从灰到绿色，不建议继续用红蓝，避免和热区语义冲突

## 12.3 中部治疗展示区

这是页面视觉核心。

### 必须包含

- 治疗台背景
- 角色主体
- 胸前异常热区
- 后背异常热区

### 设计建议

- 中部区域一定要留白，不要塞太满
- 治疗台是“主题包装”，但不要过于复杂
- 角色占中部区域视觉面积的 60% 以上
- 红蓝热区要明显，但不能比角色本体更抢
- 热区建议做轻微呼吸闪烁

### 当前关卡展示建议

- 角色姿态可用“趴着 / 歪着 / 缩着”的状态
- 胸前红区代表紧张
- 后背蓝区代表无力

## 12.4 判断提示区

这是方案 B 很关键的特色模块。

### 包含内容

- 当前判断
- 一行反馈文案

### 文案建议

- 默认判断：`当前判断：胸紧背弱`
- 错误时反馈：`这地方不是这么治的`
- 正确时反馈：`方向对了，继续恢复`

### 设计建议

- 这个区块要独立出来，像“诊断结果”
- 字号比正文稍大
- 不要做成聊天气泡，更像信息条

## 12.5 底部工具区

### 包含内容

- 工具卡片 1：筋膜球
- 工具卡片 2：弹力带
- 每张卡带一个功能说明

### 卡片文案建议

- 筋膜球
  - 主标题：`筋膜球`
  - 副标题：`放松`
- 弹力带
  - 主标题：`弹力带`
  - 副标题：`激活`

### 设计建议

- 工具区用更深一点的底色，制造游戏 HUD 感
- 卡片要看起来像可拖拽物，而不是普通按钮
- 卡片边框可以稍厚，增强“装备感”
- 拖起时卡片放大 1.05 到 1.1 倍

## 十三、交互状态设计

## 13.1 默认状态

- 红区和蓝区轻微闪烁
- 工具卡静置
- 判断区显示默认病理结论

## 13.2 工具按下状态

- 工具卡放大
- 出现轻微阴影
- 可附带高亮描边

## 13.3 正确命中状态

- 对应热区由异常色变为恢复色
- 进度条增加
- 判断区更新为积极反馈
- 播放短音效

## 13.4 错误命中状态

- 工具卡回弹
- 主区域轻微抖动
- 判断区显示错误反馈

## 13.5 通关状态

- 中部角色状态更舒展
- 恢复值满
- 结算弹层弹出

## 十四、提示弹层开发版

### 结构

```text
+----------------------------------+
| 康复提示                         |
|----------------------------------|
| 红色区域：太紧了，需要先放松     |
| 蓝色区域：太弱了，需要把它激活   |
|                                  |
| 记住一句话：松前侧，强后侧       |
|                                  |
|            [我知道了]            |
+----------------------------------+
```

### 设计建议

- 弹层宽度约为屏幕宽度的 78% 到 84%
- 背后加半透明遮罩
- 关闭方式只保留一个主按钮即可，简单直接

## 十六、推荐配色方向

MVP 阶段不必复杂，但建议统一。

### 主色

- 标题栏 / 工具栏底色：深灰蓝
- 页面背景：浅灰蓝或偏暖灰
- 分隔区域：中性灰

### 功能色

- 红区：偏橙红，避免太刺眼
- 蓝区：偏青蓝，避免过紫
- 成功反馈：绿色或金绿色
- 错误反馈：偏红橙

---

## 十七、字体与气质建议

- 标题字体偏粗，带一点漫画感
- 正文字体保持清晰可读
- 不需要过度追求“医疗专业体”
- 整体气质是“办公室梗 + 治疗小游戏”，不是医院挂号页

## 十九、最终建议

方案 B 的最佳落地方向不是“做得更专业”，而是：

- 用病例包装做主题感
- 用游戏式工具卡保证操作感
- 用简单明确的状态区降低理解成本

这会是一版非常适合 MVP 的界面基线。
