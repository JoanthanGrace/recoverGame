更新日期：2026-03-30  
目标平台：微信小游戏  
开发引擎：Cocos Creator 3.8.x  
产品阶段：最小可用版本（MVP）

---

## 一、产品概述

### 1.1 产品名称

打工人康复指南

### 1.2 产品定位

一款面向微信小游戏场景的轻量级 2D 解谜小游戏。

用户通过观察角色体态问题，识别身体前后侧不同肌肉状态，并把正确的康复工具拖拽到对应区域，完成“治疗”并获得即时正反馈。

### 1.3 一句话描述

用 30 秒到 1 分钟，治好一个因为久坐久写代码而体态失衡的打工人。

### 1.4 核心目标

- 用一个极简关卡跑通完整玩法闭环
- 验证“医学常识 + 轻解谜 + 搞笑表达”是否有吸引力
- 验证用户是否愿意完成后分享给朋友

### 1.5 目标用户

- 18-35 岁微信用户
- 经常久坐、久看电脑、对“颈椎腰椎肩颈问题”有共鸣的人
- 喜欢轻量级、可快速上手的休闲小游戏用户
- 对“打工人”“办公室亚健康”“体态梗图”有情绪共鸣的人

### 1.6 产品气质

- 轻松
- 有梗
- 上手无门槛
- 带一点点真实康复常识
- 不追求专业医疗严肃感，而追求“好懂、好玩、好分享”

## 二、MVP 目标与范围

### 2.1 MVP 要解决的问题

MVP 只验证一件事：

用户是否能快速理解“看症状 -> 选工具 -> 拖到正确部位 -> 通关分享”的核心循环，并觉得这个题材有趣。

### 2.2 MVP 必做内容

- 1 个可完整游玩的关卡
- 1 套角色体态问题设定
- 2 个可拖拽工具
- 2 个目标热区
- 正确 / 错误 / 无效操作反馈
- 健康值进度条
- 提示系统
- 通关结算弹窗
- 微信小游戏分享按钮接入位

### 2.3 MVP 不做内容

- 多关卡连续闯关
- 复杂角色成长系统
- 医疗问诊式数值系统
- 背包、商城、签到、广告、体力等商业化模块
- 社区、排行榜、好友系统
- 真人骨骼动画或高保真美术
- 大量医学知识库

### 2.4 成功标准

若用户在首次进入后：

- 能在 10 秒内理解要做什么
- 能在 1 分钟内完成关卡
- 愿意点击分享或再次挑战
- 通关后能复述一句核心知识点

则说明 MVP 基本成立。

## 三、为什么当前版本选择 Cocos

当前项目建议继续使用 Cocos Creator，而不是切换到 Phaser。

原因如下：

- 当前仓库已经是可运行的 Cocos 项目，已有基础脚本和结构
- 目标平台是微信小游戏，Cocos 的构建和适配路径更直接
- 本项目核心是 2D 场景、拖拽交互、UI 面板和简单动画反馈，和 Cocos 的能力匹配度高
- 后续如果需要继续加关卡、动画、音效、资源挂载，Cocos 编辑器工作流更顺手

结论：

MVP 阶段以尽快做出完整体验为目标，优先减少迁移成本，因此本项目技术方案定为：

`Cocos Creator 3.8.x + TypeScript + 微信小游戏发布`

## 四、核心玩法设计

### 4.1 核心体验闭环

玩家进入关卡后，完成如下循环：

1. 看到一个存在明显体态问题的角色
2. 观察角色身上的异常热区
3. 理解不同热区对应不同身体状态
4. 从底部工具栏选择一个工具并拖拽
5. 将工具放到正确的身体区域
6. 获得明确的正确或错误反馈
7. 将健康值提升到 100%
8. 进入结算页并看到一句科普解释
9. 选择分享或重玩

### 4.2 玩法规则

- 每个异常区域都对应一种明确的肌肉状态
- 每种肌肉状态只接受正确的康复工具
- 玩家需要完成全部正确操作才可通关
- 错误操作不会失败，但会给出明显反馈并让工具弹回
- 无效拖拽不扣分，只做轻反馈
- 关卡没有时间限制

### 4.3 解题逻辑

本关核心逻辑为：

- 紧张缩短的肌肉，需要放松
- 无力拉长的肌肉，需要强化

这条逻辑是整个玩法的知识基础，也决定了后续关卡可扩展性。

## 五、关卡 1：键盘侠的觉醒

### 5.1 关卡定位

这是 MVP 唯一关卡，用来验证完整玩法。

### 5.2 关卡主题

久坐伏案、长期写代码、姿势前倾，导致含胸驼背、肩颈不适的打工人。

### 5.3 病例设定

- 典型问题：上交叉综合征倾向
- 前侧问题：胸大肌紧张缩短
- 后侧问题：菱形肌 / 中下斜方肌无力拉长

### 5.4 玩家目标

使用正确工具，对两个异常区域分别进行处理，使角色恢复更健康的体态。

### 5.5 本关工具

| 工具 | 作用 | 对应操作 |
| ---- | ---- | -------- |
| 筋膜球 | 放松紧张肌肉 | 拖到胸前红色区域 |
| 弹力带 | 强化无力肌肉 | 拖到后背蓝色区域 |

### 5.6 本关热区

| 热区 | 状态 | 视觉表现 |
| ---- | ---- | -------- |
| 胸前区域 | 紧张缩短 | 红色闪烁 |
| 后背区域 | 无力拉长 | 蓝色暗淡 |

### 5.7 交互判定表

| 玩家操作 | 系统判定 | 反馈 |
| ---- | ---- | ---- |
| 筋膜球拖到胸前红区 | 正确 | 红区消失，健康值 +50，角色状态变好 |
| 弹力带拖到后背蓝区 | 正确 | 蓝区转为积极状态，健康值 +50，角色进一步恢复 |
| 筋膜球拖到后背区域 | 错误 | 错误提示，轻微抖动，工具弹回 |
| 弹力带拖到胸前区域 | 错误 | 错误提示，轻微抖动，工具弹回 |
| 工具拖到无效区域 | 无效 | 工具弹回，无惩罚 |

### 5.8 通关条件

- 两次正确操作全部完成
- 健康值达到 100%
- 弹出结算面板

## 六、页面与流程设计

## 6.1 总体流程

```text
进入游戏
-> 看到关卡主界面
-> 观察角色与热区
-> 点击提示（可选）
-> 拖拽工具进行治疗
-> 完成两次正确操作
-> 健康值满
-> 弹出结算页
-> 分享 / 再来一次 / 下一位病人（敬请期待）
```

## 6.2 页面一：主游戏页

主游戏页是玩家停留时间最长的页面，也是唯一核心操作页。

### 页面结构

#### 顶部信息区

- 关卡标题
- 健康值进度条
- 提示按钮

#### 中部主操作区

- 患者人物立绘
- 两个异常热区
- 可选的轻微动画反馈

#### 底部工具栏

- 工具 1：筋膜球
- 工具 2：弹力带

### 页面目标

- 让玩家一眼看懂“这里有病人、有异常点、有工具”
- 通过颜色和位置降低理解门槛
- 把所有操作压缩在一屏完成

## 6.3 页面二：提示弹层

提示层用于防止用户卡住。

### 触发方式

- 点击右上角提示按钮

### 显示内容

- 胸前发红：可能是紧张了，要想办法放松
- 后背发蓝：可能是太弱了，要想办法激活

提示内容要偏口语化，避免过于医学论文式表达。

## 6.4 页面三：结算弹层

当健康值达到 100% 后立即弹出。

### 结算层内容

- 通关文案
- 本关核心知识点
- 分享按钮
- 再来一次按钮
- 下一位病人按钮（MVP 中可展示“敬请期待”）

### 结算层目标

- 给出明确成就感
- 沉淀一句最关键的知识点
- 引导分享

## 七、UI 与交互设计

### 7.1 基础设计原则

- 竖屏单手操作
- 一屏完成核心体验
- 颜色含义强烈且统一
- 所有反馈要即时、明确、夸张一点

### 7.2 视觉语义

- 红色：紧张、僵硬、需要放松
- 蓝色：无力、疲软、需要强化
- 绿色 / 金色：恢复成功、积极反馈

### 7.3 关键交互

#### 工具拖拽

- 工具按下后放大一点，提示“已选中”
- 拖动时跟手
- 松手时判定落点

#### 正确反馈

- 热区颜色变化
- 角色姿态更舒展
- 健康值增加
- 弹出简短正向文案
- 音效提示

#### 错误反馈

- 屏幕或主体轻微抖动
- 工具回弹
- 弹出错误文案

#### 通关反馈

- 健康值补满
- 操作锁定
- 结算层弹出

## 八、文案设计

### 8.1 风格原则

- 简单直接
- 带一点打工人语境
- 有趣但不过度油腻
- 医学知识只说对用户有帮助的那一句

### 8.2 示例文案

#### 标题

- Lv.1 键盘侠的觉醒

#### 正确提示

- 放松到位，前胸终于不顶了
- 拉起来了，后背开始上班

#### 错误提示

- 这地方不是这么治的
- 工具不对路，再想想

#### 通关文案

- 治疗成功，打工人终于把肩背找回来了

#### 科普文案

- 长期伏案容易前侧胸肌紧、后侧背肌弱。关键思路是松前侧、强后侧。

#### 分享文案方向

- 我把一个写代码写歪的打工人救回来了，你也来试试
- 你知道胸紧背弱怎么救吗，来测测你是不是野生康复师

## 九、数值与状态设计

### 9.1 健康值

- 初始值：0
- 每次正确操作：+50
- 满值：100

### 9.2 关卡状态

| 状态 | 说明 |
| ---- | ---- |
| 未开始 | 玩家进入页面，尚未操作 |
| 进行中 | 玩家正在拖拽与尝试 |
| 已完成一项 | 任意一个正确目标完成 |
| 已通关 | 两个目标全部完成 |
| 已结算 | 结算层弹出，操作锁定 |

### 9.3 热区状态

| 状态 | 表现 |
| ---- | ---- |
| 未处理 | 显示异常颜色 |
| 已处理 | 异常消失或转正向颜色 |

## 十、音效与动效建议

MVP 阶段只需要最少的一组反馈资源。

### 10.1 音效

- 拖拽开始：轻提示
- 正确命中：舒缓或积极提示音
- 错误命中：短促错误音
- 通关：轻庆祝音

### 10.2 动效

- 红区呼吸闪烁
- 工具拖起轻微放大
- 错误操作时主区域轻微抖动
- 通关时结算层弹入

## 十一、技术实现需求

### 11.1 技术目标

保证 MVP 能在微信小游戏环境顺畅运行，开发期能快速迭代。

### 11.2 当前建议技术方案

- 引擎：Cocos Creator 3.8.x
- 语言：TypeScript
- 画面：2D UI + 简单图形或占位资源
- 发布平台：微信小游戏

### 11.3 代码层模块建议

- `GameRoot`：页面搭建、主流程控制
- `LevelConfig`：关卡配置
- `DragSystem`：拖拽逻辑与判定
- `FeedbackSystem`：文案、音效、动画反馈
- `SettlementPanel`：结算层逻辑
- `I18n / CopyConfig`：文案配置

### 11.4 资源要求

MVP 可先使用简化占位资源：

- 1 个病人角色立绘或简化轮廓图
- 2 个工具图标
- 2 个热区表现
- 1 套基础 UI
- 3 到 4 个基础音效

## 十二、埋点与验证建议

MVP 阶段建议至少统计以下行为：

- 进入游戏次数
- 点击提示次数
- 每个工具的拖拽次数
- 正确操作次数
- 错误操作次数
- 通关次数
- 分享按钮点击次数
- 再来一次点击次数

通过这些数据可以判断：

- 用户是否看得懂玩法
- 是不是卡在某个交互点
- 分享意愿是否存在

## 十三、风险与应对

### 13.1 风险：题材理解门槛偏高

用户可能不知道“放松”和“强化”分别对应什么工具。

应对：

- 提示按钮文案更口语化
- 工具标签更明确
- 初期关卡只给两个工具，降低选择成本

### 13.2 风险：医学感过强导致不够轻松

应对：

- 文案保持打工人语境
- 角色视觉偏漫画化
- 科普只保留一句结论

### 13.3 风险：关卡过短导致内容不足

应对：

- MVP 先验证核心玩法
- 后续扩展用“下一位病人”承接新关卡

### 13.4 风险：技术切换造成返工

应对：

- MVP 阶段不切技术栈
- 直接沿用当前 Cocos 项目继续迭代

## 十五、最终结论

《打工人康复指南》的 MVP 本质上不是做一个复杂游戏，而是做一个非常顺、非常易懂、非常容易分享的核心体验样板。

当前最重要的不是功能堆叠，而是把这条链路做好：

`看懂问题 -> 选对工具 -> 拖对位置 -> 获得成就感 -> 愿意分享`

只要这一条链路顺了，这个项目就值得继续扩展成多病人、多关卡的小游戏产品。
